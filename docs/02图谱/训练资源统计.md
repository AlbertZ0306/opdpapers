# OPD 论文训练资源使用统计

更新时间：2026-06-24

来源：对 `papers/llm_opd_methods/` 下 **110 篇** PDF 用 `pypdf` 抽取"实现细节/实验设置/附录"中的资源相关行，再人工归纳。统计字段：**硬件配置、训练时长、训练规模设置、是否使用 LoRA、训练框架**。

## 统计口径与重要提醒

- 仅统计 `llm_opd_methods/`（核心 LLM OPD/OPSD 方法与分析论文）；`model_tech_reports/`、`multimodal_and_adjacent/`、`local_originals/` 未纳入本轮。
- **大量字段为"未明确"是真实现象**：OPD 论文（尤其 2026-05+ 增量的快节奏 arXiv 文）普遍**不报告硬件型号/数量与 GPU-hours**，常只给 step 数或 epoch 数。"未明确"= 论文正文/附录未报告，非遗漏。
- LoRA 一列：`LoRA`=明确用低秩适配；`全参`=明确 full fine-tuning / full-parameter / FSDP full-shard；`未明确`=未交代 PEFT 还是全参。注意有的论文**仅 teacher 或某阶段**用 LoRA。
- 个别论文 PDF 文本抽取不完整（无 poppler 环境做深读），其稀疏字段以"未明确"标注，最稀疏的是 `2605.06387`、`2606.09471`、`2605.28396`。
- 训练时长口径混杂（GPU-hours / wall-clock / step / epoch），按论文原文给出的单位记录。

## 一、聚合统计（关键发现）

### 1.1 硬件配置

- **主流规模 = 单节点 8×GPU**：绝大多数明确报告硬件的论文用 1 node × 8 卡（A100 / H100 / H800 / H20 / A800 / B200 各家都有）。
- **多节点（≥32 卡）少数**：`2602.22495`(64×H200)、`2605.26844`(64×H800)、`2605.27115`(32×H100)、`2605.30833`(32×H20)、`2606.18810`(8 节点×8 A100=64)。
- **单张/双张卡的诊断或效率类**：`2604.04461`(1×A6000)、`2605.22731`(1×RTX3090)、`2605.28303`/`2605.30251`(1×RTX4090)、`2605.16826`(1×H100)、`2606.00172` CAST(2×A800)、`2601.07155`/`2602.04942`/`2606.02530`(2×H100)。这类多是小模型受控实验或主打"低成本"。
- GPU 型号谱：H100 / H200 / A100 / A800 / H800 / H20 / B200 / Blackwell RTX PRO 6000 / GH200 / A6000 / RTX 4090 / RTX 3090 都出现，**Hopper 系（H100/H200/H800/H20）与 A100/A800 为主力**。

### 1.2 训练时长

- **最常见表述是 step 或 epoch**：OPD/OPSD 蒸馏阶段普遍 **100–500 step**（少数 1000–1500），或 **1–3 epoch**。
- **报告 GPU-hours 的少数（值得参考的量级）**：`2604.13010` Lightning OPD 20–30 GPU·h、`2605.08737` ~12 GPU·h、`2603.11137` ~200–312 GPU·h、`2605.27115` ~224 GPU·h、`2605.30251` ~132 GPU·h、`2605.23493` ~1020 H100·h、`2605.30070` ~720 B200·h、`2606.00147` ~1000 GPU·h、`2606.04071` ~910 H200·h；诊断套件 `2605.10889` ~72 A100-days；评测开销 `2606.09304` ~7000 A100·h。
- **wall-clock 加速是高频卖点**：多篇报告自家方法比 vanilla OPD/GRPO 快（`2604.13010` 4×、`2605.11739` >3×、`2605.27028` 24×、`2606.08432` 省~60%、`2606.15912` ~2.5h vs ~10h、`2606.17199` 省 59.2%）。

### 1.3 是否使用 LoRA（重要）

- **明确用 LoRA 的约 24–25 篇**（约 22%），且高度集中在**自蒸馏/OPSD 与效率导向**工作；典型配置 **rank 64 / α 128**（也有 r8、r16、r128）。代表：`2601.18734`、`2605.07865`(vOPD)、`2606.00172`(CAST)、`2605.06597`、`2605.21606`、`2605.22675`、`2606.17628`、`2606.18810`、`2606.19327` 等。
- **明确全参（full FT / FSDP full-shard）的约 14–15 篇**：`2604.03873`、`2604.12002`、`2604.13016`、`2605.02943`、`2605.08737`、`2605.12483`、`2605.19433`、`2605.22263`、`2605.23493`、`2606.03089`、`2606.04036`、`2606.09304`、`2606.11627`、`2606.11709`。
- **其余约 70 篇未明确 PEFT 还是全参**——这是该领域报告习惯上的一个普遍缺口。
- 对 P5 的启示：**LoRA 在本领域是被广泛接受的、低成本的合法做法**（连 vOPD、CAST 这些我们要对照的工作都用 LoRA），P5 的 kill test 用 LoRA + 2×A800 量级是站得住的。

### 1.4 训练框架

- **verl / VeRL 占主导**（数十篇），常配 **vLLM/SGLang**（rollout 生成）与 **FSDP**。
- 其他出现：DeepSpeed ZeRO-2/3、TRL、Megatron（Miles 框架）、slime、LLaMA-Factory、ms-swift、NeMo-RL、PEFT/Accelerate、RFT、自定义。
- teacher 常作为**独立 vLLM/SGLang 服务**或单卡进程，与 student 训练解耦（`2604.10688`、`2605.03677`、`2605.23493`、`2606.09456`、`2606.10385`）。

### 1.5 训练规模常见缺省值（可作 P5 选型参考）

- **batch**：32–256（少数 512–1024）；prompt batch 常 32–128。
- **每 prompt rollout 数 G**：**8 最常见**（也有 4 / 12 / 16）。
- **学习率**：OPD/RL 阶段 **1e-6 最常见**（范围 5e-7 ~ 1e-5）；SFT/LoRA 阶段偏高（5e-6 ~ 1e-4）。
- **max response 长度**：4k–32k（长程任务到 38k、甚至 192k）；max prompt 常 1024–4096。
- **步数/epoch**：蒸馏 100–500 step 或 1–3 epoch。

> **与 P5 的直接关联**：上述缺省值正是 P5 实验设计里 Qwen3-1.7B + DAPO-Math、G=8、lr≈1e-6、batch 32–128、100–300 step 选型的文献依据；本表证明这套配置在单节点 8 卡（甚至 2×A800 + LoRA，如 CAST）即可复现，P5 的最小配置可行。

### 3.6 专项结论（资源视角）

1. **可直接复用的配方就在最相关的两篇里**：**CAST（2×A800 + LoRA r64α128 + 300 步 + 8 completions/prompt + lr1e-6）** 与 **vOPD（1×H200/A100 + LoRA + 2–6h）** 的配置，几乎就是 P5 的 B 系列基线与最小配置——P5 可**逐项照搬**，无需另调。
2. **C-OPD-self 的现成 recipe = OPSD Compresses**：4×H100、batch32、lr1e-6 常数无 warmup、100 步 ~6h、eval 30k-token temp0.6——P5 的 self-teacher OPD 臂可直接对齐此设置以保证可比。
3. **strong-to-weak（E2 δ<0）参 ExOPD / RLSD**：batch128–256、8–16 rollouts/prompt、lr1e-6、teacher 8B–30B；RLSD 的"teacher 每 10 步 sync"也是 P5 self-teacher 刷新可借鉴的口径。
4. **资源层面的差异化（写进卖点）**：最强竞争对手 **Unmasking OPD 花 ~72 A100-days、每问 45K–200K rollouts** 才做出诊断；而 **P5 的 Tier 0 协方差代理 + ρ 近乎免费**（复用一条 GRPO 的 rollout + 纯前向）。这是 P5 相对该诊断的**成本碾压**式区分点。
5. **低成本诊断有先例**：`2605.22731`（1×RTX 3090 + LoRA）与 `2605.16826`（1×H100）证明"单卡 LoRA 诊断/归因"足以发表——直接支撑 P5 kill test 用 Qwen3-1.7B + 少量卡 + LoRA 的可行性与正当性。
6. **这批最相关论文的算力区间**：除两篇诊断（72 A100-days；1000 步 30h 级）外，主体落在**单节点 ≤8 卡、100–500 步、1–6h~1 天**；P5 复现"GRPO/OPD-self/1×VR ×≥3 seed"完全在此区间内。

## 四、对 P5 的可操作结论

1. **规模可控**：本领域主流即"单节点 8×GPU + 100–500 step + G=8 + lr≈1e-6"，与 P5 设计完全一致；连 vOPD、CAST 等核心对照都在 1–2 张卡 + LoRA 上完成，**P5 kill test 的最小配置（Qwen3-1.7B + 2×A800 + LoRA）有充分先例**。
2. **LoRA 合法且常见**：约 22% 论文明确用 LoRA（含 vOPD/CAST），P5 的 B 系列基线与 OPD/GRPO 对照可统一用 LoRA 降成本，不损可比性。
3. **公平性提醒**：本领域**普遍不报硬件/GPU-hours**，P5 若主打"成本/方差缩减"叙事，**显式报告算力对齐口径（见实验设计 §5、P−5）反而是差异化加分项**。
4. **框架默认 verl**：训练器选 verl + vLLM/FSDP 与文献主流一致（P5 Step 6 直接复用 `/data1/zhaoenyue/experiments/verl`）。
